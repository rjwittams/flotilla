Issue System — Data Flow Analysis

How issues bypass the normal provider→snapshot pipeline and why it creates architectural tension.

A. What does the default provider do?

Normal Providers (checkouts, PRs, sessions...)

refresh loop runs → provider.list_*() fetches ALL items → stores in ProviderData fields → correlate() builds work items → snapshot broadcast One source of truth. Complete data. Every snapshot is self-contained.

Issue Provider

refresh loop does NOT fetch issues → issues come from IssueCache (side channel) → cache populated by on-demand paging → inject_issues() replaces ProviderData.issues → correlate() builds work items → snapshot broadcast Two sources of truth. Partial data. Snapshot depends on cache state.

B. The Issue Data Pipeline

TUI first render
SetIssueViewport { visible_count: N } → ensure_issues_cached(N * 2) Fetches ~2 pages (100 issues) via GitHub API. Stores in IssueCache.
↓
User scrolls down
FetchMoreIssues { desired_count } → ensure_issues_cached(current + 50) Fetches next page. IssueCache.next_page increments. has_more updated.
↓
User searches
SearchIssues { query } → tracker.search_issues() Results stored in RepoState.search_results (separate from cache). inject_issues() prefers search_results over cache.
↓
Every poll cycle
(100ms)
poll_snapshots() calls inject_issues_from_entries() to splice cached/searched issues into ProviderData before correlation. Issues in ProviderData from refresh are always replaced.
↓
Every 30s
refresh_issues_incremental() calls list_issues_changed_since(). Upserts updated issues, evicts closed ones. Escalates to full re-fetch if too many changes.
↓
On PR correlation
fetch_missing_linked_issues() finds issues referenced by PRs but not in cache. Fetches individually and pins them (survive eviction).

C. The Snapshot Integrity Problem

Issues break the snapshot model

Every other provider produces complete, self-contained data during the refresh cycle. The snapshot is a pure function of provider output. But issues are different:

  • The refresh cycle produces no issue data at all — ProviderData.issues is empty from providers
  • Issues are injected after the fact by inject_issues(), replacing whatever was in ProviderData.issues
  • The injected data depends on cache state (how far the user has scrolled) and search state (active query)
  • Two clients connected to the same daemon see different issue data if one has scrolled further
  • The cache is a monotonically growing window — it only adds pages, never shrinks (except on escalation reset)

Data Shape at Each Stage

─── After provider refresh ─── ProviderData.issues = {} ← always empty, providers don't fetch issues ─── After inject_issues() ─── ProviderData.issues = { "#1": Issue { title: "Bug A", ... }, ← from IssueCache (first 100) "#2": Issue { title: "Feature B", ... }, ... "#100": Issue { title: "Cleanup Z", ... }, } ← arbitrary subset determined by how many pages the TUI has fetched ← if user searches, this is replaced with search results instead ─── Snapshot metadata (separate from ProviderData) ─── RepoSnapshot.issue_total = Some(847) ← from GitHub API RepoSnapshot.issue_has_more = true ← more pages available RepoSnapshot.issue_search_results = None ← or Some(...) during search

How Other Providers Compare

Checkouts / PRs / Sessions

Provider fetches ALL items → stored directly in ProviderData → snapshot is complete → no caching layer needed → no user-dependent state → delta computation is clean

Issues

Provider fetches NOTHING → IssueCache populated on-demand → inject_issues() splices into ProviderData → snapshot depends on scroll position → search results overlay cache → delta sees issue churn on every scroll → 3 extra fields on RepoSnapshot → 4 special command handlers → per-repo mutex for paging → pinned set for linked issues

Why Issues Work Differently

The Scale Problem

A repo might have 10 checkouts, 20 PRs, 5 agent sessions — but thousands of issues. Fetching all issues on every refresh is impractical. The pagination/caching design exists because issues are the only provider data type where "fetch everything" doesn't scale.

The Consequences

  • inject_issues() — unique to issues, splices cache into provider data
  • IssueCache — unique to issues, no other provider has a cache layer
  • issue_total / issue_has_more — snapshot metadata that only exists for issues
  • search_results overlay — UI state leaking into the data model
  • Pinning — linked issues must survive cache operations
  • Incremental refresh — since-based polling unique to issues
  • Fetch mutex — serialization unique to issues

Architectural Tension

The snapshot model assumes ProviderData is authoritative and complete. Issues violate this by making ProviderData.issues a view window into a larger dataset. The delta system then computes diffs on this partial, user-dependent data — meaning deltas carry issue churn that has nothing to do with actual state changes.

The TUI treats issue_total/issue_has_more as hints for infinite scroll, but these fields travel through the entire snapshot→delta→replay pipeline even though they're really UI pagination state.

The Core Question

Should issues be in ProviderData at all?

Issues serve two distinct purposes:

  • Linked issues — referenced by PRs/checkouts via AssociationKey. These are small in number and must participate in correlation. They belong in the snapshot.
  • Issue list — the scrollable table of open issues. This is a paginated view of an external dataset. It doesn't participate in correlation (standalone issues become work items, but they're not correlated with anything).

Possible direction: linked issues stay in ProviderData (pinned, always fetched). The issue list becomes a separate query/pagination channel that doesn't flow through the snapshot pipeline. The TUI manages its own issue list state, fetching pages directly. The snapshot only carries linked issues.
